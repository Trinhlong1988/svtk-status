## Detailed evidence per round

### Round 111 — Validator heartbeat file presence + pipeline refactor

**Bug:** Build wrote heartbeat AFTER validator via `write_lead_heartbeat_completion(status)`. Validator could never check it. LEAD heartbeat-watcher could miss a build if heartbeat write failed silently.

**Fix (compound):**
1. New `write_lead_heartbeat_pre()` writes BUILDING placeholder heartbeat before alerts/validator:
   ```python
   {"worker": "cmd-npc", "alive": True, "score": null, "status": "BUILDING"}
   ```
2. `main()` calls it after `sweep_old_npc_artifacts()` and before alert chain.
3. Validator check #71 `heartbeat_file_present_R111` confirms file exists at `cmd-lead/heartbeats/cmd-npc-{TS}.json`.
4. `write_lead_heartbeat_completion(status)` post-validator overwrites placeholder with final score/count.

**Post-fix:** File present at validator time → check #71 PASS.

---

### Round 112 — Validator completion file presence

**Bug:** Same lifecycle issue as R111 for completion file.

**Fix (compound):**
1. `write_lead_heartbeat_pre()` also writes `BUILDING-{fix_id}-{TS}.json` to `cmd-lead/completions/`.
2. Validator check #72 `completion_file_present_R112` globs `*npc*{TS}*.json` (matches BUILDING placeholder).
3. `write_lead_heartbeat_completion()` deletes BUILDING placeholder + writes final `PASS-{fix_id}-{TS}.json`.

**Post-fix:** Completion file present at validator + finalized post-validator with real result.

### Round 113 — Validator foundation v2.8.0 presence

**Bug:** Build calls `verify_foundation()` which `sys.exit(99)` if foundation missing — but validator did not formally record this as a check passable to LEAD inspection.

**Fix:** Validator check #73 `foundation_v280_present_R113` checks file existence + records hash prefix in evidence:
```python
f28_path = REPO_DIR / "foundation" / "SVTK_FOUNDATION_v2.8.0.md"
f28_hash = hashlib.sha256(f28_path.read_bytes()).hexdigest()[:16]
```

**Post-fix:** Check passes; LEAD can confirm exact foundation hash used by inspecting validation.json now.

### Round 116 — validation.json `checks` detail enrichment

**Bug:** Pre-fix `validation.json` only exposed `errors` array — downstream LEAD/QA could not inspect WHICH checks passed or see evidence for PASS checks. Made debugging "near-pass" runs harder.

**Fix:** `validator()` now writes:
```python
"checks": [
    {"name": "schema_uniformity_R1", "ok": true, "evidence": {...}},
    {"name": "uuid_R74A", "ok": true, "evidence": {...}},
    ...
]
```
75-entry array with name + ok + evidence per check.

**Post-fix:** LEAD review tooling can inspect entire validator state without re-running build.

### Round 117 — Foundation version mismatch alert

**Bug:** Brief locks `EXPECTED_FOUNDATION_HASH_V26 = "2e6e8c23..."` for SVTK_FOUNDATION_v2.6.0. Repo only ships v2.8.0 (`ab1b4eb2...`). `verify_foundation()` accepts v2.8.0 silently with a foundation_note in status. Adjacent CMDs (CMD_DB, CMD_BOSS, CMD_QUEST, etc.) likely have same expectation — silent drift across the project.

**Fix:** `alert_foundation_version_mismatch()` MED severity. Reports brief expected version + hash, repo actual version + hash, remediation note for LEAD to canonicalize foundation version across all CMDs OR re-issue briefs with v2.8.0 hash.

**Post-fix:** Alert file ships per build. LEAD inbox now visible for cross-CMD reconciliation.
