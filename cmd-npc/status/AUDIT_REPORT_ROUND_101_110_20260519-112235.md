## Detailed evidence per round

### Round 101 — Validator sha256 sidecar match

**Bug:** Each registry jsonl has a `.sha256` sidecar but no validator confirmed they match content. Silent file corruption (e.g., partial write, manual edit) could ship without detection.

**Fix:** Validator check #66 `sha256_sidecar_match_R101`:
```python
for s_name in ("main","side","lore","generated","full"):
    p = out_reg / f"npc_{s_name}.jsonl"
    sha_p = out_reg / f"npc_{s_name}.jsonl.sha256"
    declared = sha_p.read_text(encoding="utf-8").strip().split()[0]
    computed = hashlib.sha256(p.read_bytes()).hexdigest()
    if declared != computed: sha_mismatch.append(s_name)
```

**Post-fix:** 5/5 match. Probe P1 confirms: main bca1e67..., side 7cc51982..., lore 763f4761..., generated 003539d0..., full 045abf3a... all match.

### Round 103 — Validator SQL CHECK enum match

**Bug:** SQL DDL has CHECK constraints `element IN (...)` and `npc_type IN (...)`. If JSON gen adds a new element/type without DDL update, ingest would fail silently in production DB. Validator did not enforce json ⊆ sql.

**Fix:** Validator check #68 parses SQL CHECK lists with regex, compares to JSON values:
- element: {kim, mộc, thủy, hỏa, thổ, tâm} ⊆ SQL allowed ✓
- npc_type: 10 types ⊆ SQL allowed ✓

**Post-fix:** Match (P5 + P19 confirm).
